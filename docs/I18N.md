## Getting Started

### Using Translations in Components

```typescript
import { useTranslation } from 'react-i18next';

function MyComponent() {
  const { t } = useTranslation();
  
  return (
    <div>
      <h1>{t('common.welcome')}</h1>
      <p>{t('auth.loginSuccess')}</p>
    </div>
  );
}
```

### Basic Translation

```typescript
// Simple translation
t('common.save') // → "Save" (EN) / "Guardar" (ES)

// Nested keys (dot notation)
t('auth.login') // → "Login"
t('users.createUser') // → "Create User"
```

## Translation Keys Convention

### Namespace Organization

Organize keys by feature/page:

```plaintext
common.*         - Shared UI elements (buttons, labels)
auth.*           - Authentication (login, register, logout)
navigation.*     - Main navigation items
dashboard.*      - Dashboard page
users.*          - Users management
roles.*          - Roles management
profile.*        - User profile
settings.*       - Settings page
apiKeys.*        - API Keys management
notifications.*  - Notifications
sessions.*       - Session management
mfa.*            - Two-factor authentication
tenants.*        - Tenant management
search.*         - Search functionality
oauth.*          - OAuth/SSO flows
audit.*          - Audit logs
validation.*     - Form validation messages
errors.*         - Error messages
```

### Naming Patterns

| Pattern | Example | Usage |
| ------- | ------- | ----- |
| **Actions** | `createUser`, `deleteRole` | Buttons, links |
| **Labels** | `firstName`, `email` | Form fields |
| **Messages** | `createSuccess`, `deleteError` | Alerts, toasts |
| **Descriptions** | `subtitle`, `description` | Help text |
| **Titles** | `title`, `modalTitle` | Headings, modal headers |
| **Placeholders** | `searchPlaceholder`, `enterEmail` | Input placeholders |
| **Validation** | `required`, `emailInvalid` | Error messages |

### Examples

```json
{
  "users": {
    // Page structure
    "title": "Users",
    "subtitle": "Manage your organization's users",
    
    // Actions (verbs)
    "createUser": "Create User",
    "editUser": "Edit User",
    "deleteUser": "Delete User",
    
    // Labels (nouns)
    "firstName": "First Name",
    "lastName": "Last Name",
    "email": "Email",
    
    // Messages (past tense + status)
    "createSuccess": "User created successfully",
    "createError": "Failed to create user",
    "deleteConfirm": "Are you sure you want to delete this user?",
    
    // Placeholders
    "searchPlaceholder": "Search users by name or email...",
    
    // Empty states
    "noUsersFound": "No users found",
    "tryDifferentSearch": "Try a different search term"
  }
}
```

## Advanced Features

### 1. Parameterization (Interpolation)

Pass dynamic values to translations:

**Translation File:**

```json
{
  "users": {
    "welcome": "Welcome back, {{name}}!",
    "itemCount": "Showing {{count}} of {{total}} items"
  }
}
```

**Component:**

```typescript
t('users.welcome', { name: 'John' })
// → "Welcome back, John!"

t('users.itemCount', { count: 10, total: 50 })
// → "Showing 10 of 50 items"
```

### 2. Pluralization

Handle singular/plural forms:

**Translation File:**

```json
{
  "notifications": {
    "unreadCount": "You have {{count}} unread notification",
    "unreadCount_plural": "You have {{count}} unread notifications"
  }
}
```

**Component:**

```typescript
t('notifications.unreadCount', { count: 1 })
// → "You have 1 unread notification"

t('notifications.unreadCount', { count: 5 })
// → "You have 5 unread notifications"
```

### 3. Dynamic Keys with Template Literals

For dynamic translations:

**Translation File:**

```json
{
  "tenants": {
    "plans": {
      "free": "Free",
      "starter": "Starter",
      "professional": "Professional",
      "enterprise": "Enterprise"
    }
  }
}
```

**Component:**

```typescript
const plan = "professional";
t(`tenants.plans.${plan}`)
// → "Professional"
```

### 4. Nested Objects

Use nested structures for complex UIs:

```json
{
  "settings": {
    "tabs": {
      "profile": "Profile",
      "security": "Security",
      "connections": "Connections"
    },
    "notifications": {
      "email": {
        "title": "Email Notifications",
        "description": "Receive email updates"
      }
    }
  }
}
```

```typescript
t('settings.tabs.profile') // → "Profile"
t('settings.notifications.email.title') // → "Email Notifications"
```

### 5. Default Values

Provide fallback for missing keys:

```typescript
t('missing.key', 'Default text if key not found')
// → "Default text if key not found"
```

### 6. HTML in Translations (Use with Caution)

For translations with formatting:

```typescript
import { Trans } from 'react-i18next';

<Trans i18nKey="auth.agreeToTerms">
  I agree to the <a href="/terms">Terms of Service</a> and 
  <a href="/privacy">Privacy Policy</a>
</Trans>
```

## Adding New Languages

### Step 1: Create Translation File

Create `frontend/src/i18n/locales/[lang].json`:

```bash
# Example: Adding Italian (IT)
cd frontend/src/i18n/locales
cp en.json it.json
```

### Step 2: Translate All Keys

Edit `it.json` and translate all ~700 keys from English to Italian.

**Tips:**

- Use a professional translator or translation service
- Maintain the same JSON structure
- Keep parameterization syntax: `{{count}}`, `{{name}}`
- Preserve HTML tags if present
- Test pluralization rules for the language

### Step 3: Update i18n Configuration

Edit `frontend/src/i18n/index.ts`:

```typescript
export const SUPPORTED_LANGUAGES = [
  { code: 'en', name: 'English', flag: '🇺🇸' },
  { code: 'es', name: 'Español', flag: '🇪🇸' },
  { code: 'pt', name: 'Português', flag: '🇧🇷' },
  { code: 'it', name: 'Italiano', flag: '🇮🇹' }, // ← Add new language
  // Disabled: { code: 'fr', name: 'Français', flag: '🇫🇷' },
  // Disabled: { code: 'de', name: 'Deutsch', flag: '🇩🇪' },
] as const;
```

**Note:** Don't forget to add the new language to the `preload` array in the backend configuration:

```typescript
backend: {
  loadPath: '/locales/{{lng}}.json',
  preload: ['en', 'es', 'pt', 'it'], // ← Add new language
}
```

### Step 4: Copy to Public Directory

```bash
cd frontend
cp src/i18n/locales/it.json public/locales/
```

### Step 5: Test

1. Change language in settings
2. Navigate through all pages
3. Verify all text displays correctly
4. Test parameterized translations
5. Check pluralization works

## Best Practices

### ✅ Do's

1. **Use Descriptive Keys**

   ```typescript
   // ✅ Good
   t('users.deleteConfirmMessage')
   
   // ❌ Bad
   t('users.msg1')
   ```

2. **Keep Keys Organized by Feature**

   ```json
   {
     "users": { /* all user-related keys */ },
     "roles": { /* all role-related keys */ }
   }
   ```

3. **Use Parameterization for Dynamic Content**

   ```typescript
   // ✅ Good
   t('users.welcome', { name: user.name })
   
   // ❌ Bad
   `Welcome, ${user.name}!`
   ```

4. **Translate All User-Facing Text**
   - Error messages
   - Success messages
   - Placeholders
   - Help text
   - Button labels
   - Modal titles

5. **Maintain Consistent Terminology**
   - Use the same translation for the same concept
   - Create a glossary for technical terms

6. **Test in All Languages**
   - Verify layout doesn't break with longer translations
   - Check right-to-left languages if supported
   - Test pluralization edge cases

### ❌ Don'ts

1. **Don't Hardcode Text**

   ```typescript
   // ❌ Bad
   <button>Save</button>
   
   // ✅ Good
   <button>{t('common.save')}</button>
   ```

2. **Don't Translate Developer-Facing Content**
   - Console logs
   - Developer error messages
   - Code comments
   - API endpoints

3. **Don't Use Translation Keys as Defaults**

   ```typescript
   // ❌ Bad
   t('users.title') || 'Users'
   
   // ✅ Good
   t('users.title') // Already has fallback to EN
   ```

4. **Don't Mix Languages in Keys**

   ```json
   // ❌ Bad
   {
     "usuarios": "Users"
   }
   
   // ✅ Good - Keys always in English
   {
     "users": "Users"
   }
   ```

## Performance Optimization

### Current Optimizations

#### 1. Lazy Loading (Implemented)

Translations are loaded on-demand via HTTP Backend:

```typescript
// frontend/src/i18n/index.ts
i18n.use(HttpBackend).init({
  backend: {
    loadPath: '/locales/{{lng}}.json',
  },
  load: 'currentOnly', // Only load current language
});
```

**Benefits:**

- ✅ Reduced initial bundle size (~150KB+ savings)
- ✅ Faster initial page load
- ✅ Only loads active language
- ✅ Automatic code splitting

#### 2. Preloading Critical Languages

```typescript
backend: {
  preload: ['en', 'es'], // Most used languages
}
```

#### 3. Single Namespace Strategy

Uses one namespace (`translation`) for simplicity:

- Easier to manage
- Faster lookups
- No namespace switching overhead

### Bundle Size Impact

| Strategy | Bundle Size | Load Time |
| -------- | ----------- | --------- |
| **Before (Static Import)** | ~850KB | 1.2s |
| **After (Lazy Loading)** | ~700KB | 0.8s |
| **Savings** | **~150KB** | **33% faster** |

### Future Optimizations (Optional)

#### Multi-Namespace Strategy

For larger applications, split by feature:

```typescript
// Split into namespaces
i18n.init({
  ns: ['common', 'auth', 'users', 'settings'],
  defaultNS: 'common',
});

// Use in components
const { t } = useTranslation(['users', 'common']);
t('users:title'); // From users namespace
t('common:save'); // From common namespace
```

#### Route-Based Code Splitting

Load translations per route:

```typescript
// Example: Load only needed namespaces per route
const UsersPage = lazy(() => 
  import('./pages/UsersPage').then(module => {
    i18n.loadNamespaces('users');
    return module;
  })
);
```

## Troubleshooting

### Common Issues

#### 1. Translation Not Showing

**Problem:** `t('some.key')` shows key instead of translation

**Solutions:**

- ✅ Check key exists in translation files
- ✅ Verify JSON syntax is valid (no trailing commas)
- ✅ Ensure language file is loaded (check Network tab)
- ✅ Check for typos in key name
- ✅ Verify namespace is correct

#### 2. Interpolation Not Working

**Problem:** `{{name}}` appears literally in UI

**Solutions:**

```typescript
// ❌ Wrong
t('welcome', name: 'John')

// ✅ Correct
t('welcome', { name: 'John' })
```

#### 3. Pluralization Not Working

**Problem:** Always shows singular form

**Solutions:**

- Ensure `_plural` suffix is added to key
- Pass `count` in options object
- Check language-specific plural rules

```typescript
// Translation
{
  "items": "{{count}} item",
  "items_plural": "{{count}} items"
}

// Usage
t('items', { count: 1 })  // → "1 item"
t('items', { count: 5 })  // → "5 items"
```

#### 4. Language Not Switching

**Problem:** UI doesn't update after language change

**Solutions:**

```typescript
import { useTranslation } from 'react-i18next';

function LanguageSwitcher() {
  const { i18n } = useTranslation();
  
  const changeLanguage = (lng: string) => {
    i18n.changeLanguage(lng); // This triggers re-render
  };
  
  return (
    <select onChange={(e) => changeLanguage(e.target.value)}>
      <option value="en">English</option>
      <option value="es">Español</option>
    </select>
  );
}
```

#### 5. Missing Translation Warning

**Problem:** Console shows "key not found" warnings

**Solutions:**

- Add missing key to translation files
- Provide default value: `t('missing.key', 'Default text')`
- Set `fallbackLng: 'en'` in config (already set)

#### 6. Build Errors with Lazy Loading

**Problem:** Production build fails

**Solutions:**

- Ensure `public/locales/` directory exists
- Copy translation files to public: `cp src/i18n/locales/*.json public/locales/`
- Add to `.gitignore` if needed
- Verify Vite config serves `public/` correctly

### Automated Testing

```typescript
// Example: Test component with translations
import { render, screen } from '@testing-library/react';
import { I18nextProvider } from 'react-i18next';
import i18n from './i18n';
import MyComponent from './MyComponent';

test('renders translated text', async () => {
  await i18n.changeLanguage('es');
  
  render(
    <I18nextProvider i18n={i18n}>
      <MyComponent />
    </I18nextProvider>
  );
  
  expect(screen.getByText('Usuarios')).toBeInTheDocument();
});
```
